# agent-remnote 搜索策略分层（草案）：插件候选集 + DB 精筛

> 目标：在不牺牲“可用性与可解释性”的前提下，把检索链路做成**可控、可兜底、可协作**的两阶段流程，避免 Agent 因为一次“大查询”卡死在本地 SQLite 上。
>
> 状态：草案（先定边界与兜底；实现与协议落地后再提升为 SSoT）。
>
> 对应需求与任务：`specs/005-search-safety/spec.md`、`specs/005-search-safety/tasks.md`（依赖：`specs/003-ws-identity/spec.md`）

## 核心结论（无歧义边界）

- **插件侧（前端）只负责“快速候选集生成”**：给一个关键词/短语，在 RemNote 内部搜索语义下返回 Top‑K 候选 Rem（默认 `K=20`），并在 `3s` 内返回；超时/失败立刻回退。
- **后端（外部）只负责“确定性精筛/分页/展开”**：对候选 RemId 做结构化过滤、路径补全、分页展开、引用/属性解析等；并提供更重的只读 DB 查询（目标 `≤30s`，需要子进程/worker 才能真正超时中断）。
- **两者不是强制绑定**：插件候选集用于“探索期加速”，但任何时候都允许直接走后端 DB（例如插件不在线、用户给了明确定位线索、或任务天然需要结构化遍历）。

## 为什么需要“候选集 + 精筛”

RemNote 桌面端内部有自己的搜索索引与排序语义；而外部环境直接读 `remnote.db` 时：

- `better-sqlite3` 为同步调用，**慢查询会阻塞 Node 进程**，无法在主线程可靠地“中断”；
- FTS（如 `remsContents`）在外部 sqlite 环境里可能因 tokenizer 不可用而失效；
- 一些 JSON 条件（尤其 `instr(json_extract(...))`、属性/引用反查）天然容易退化为全表扫描。

因此将“探索式定位”放到插件内部做 Top‑K，再把重活交给后端做确定性处理，是最稳的协作形态。

## 何时需要插件做“快速候选集生成”

满足以下任一条件时，优先考虑插件候选集（但仍受预算约束）：

- 用户/Agent 有**可落地的词法线索**：关键词、标题片段、别名片段、少量关键短语、英文单词组合等（本质是字符串检索，不是语义 RAG）。
- 需要 RemNote **UI 的排序语义**快速找到“最可能的入口 Rem”（relevance/rank），然后再深入展开。
- 希望在一个上下文范围里先找入口（例如“当前页面/某个页面内先搜”），可用 `searchContextRem` 收敛范围。
- 外部 DB 侧 FTS 不可用或风险高（预计会卡住/扫全表）时，先用插件拿种子。

不适合直接走插件候选集的情况（应先补充线索或直接走 DB 的结构化查询）：

- 用户只说“找我之前写过的那条……”但**没有任何关键词/范围线索**；此时插件搜索不会“理解意图”，只会做字符串检索。

## 插件 API 能力与“命中附近文本”

`@remnote/plugin-sdk` 提供：`plugin.search.search(queryText, searchContextRem?, { numResults, filterOnlyConcepts })`。

- 返回类型是 `PluginRem[]`，**没有专门的 snippet/highlight 字段**。
- 但 `PluginRem`（`RemObject`）通常携带 `text` 与可选 `backText`（富文本）。

因此“命中附近文本”建议按以下优先级生成（返回给 Agent 用于判别相关性）：

1. 若 `PluginRem.text/backText` 已足够小：直接转成纯文本并截断为 `maxPreviewChars`。
2. 若需要更像“命中片段”：将查询拆成 token（按空格/标点；中文可按短语优先），在纯文本中定位首个命中位置，截取窗口（例如 `±80` 字符）生成 `snippet`；找不到命中则回退到“开头预览”。

无论哪种方式，都应携带 `truncated=true/false`，避免误导 Agent 以为拿到的是全文。

## 预算与兜底策略（必须）

### 插件候选集（前端）

- `timeoutMs`：默认 `3000`，最大不超过 `5000`（超过即视为不可用，直接回退）。
- `limit`：默认 `20`；允许显式提高（建议上限 `100`），但必须在响应中给出“为何需要提高”的建议说明。
- `payload`：返回字段应精简（`remId + title + snippet/preview + timestamps(可选)`）；对富文本必须做长度裁剪并标记 `truncated`。
- **禁止**在候选集阶段做深度展开（子树、引用链、属性反查、全文 dump）。

失败/超时时，响应应提供建议型 next actions（不强制），例如：

- “缩短/拆分关键词；优先用 1–3 个核心词”
- “提供上下文 Rem（例如 pageRemId）以收敛范围”
- “改走 DB 搜索并限制 timeRange（例如最近 30 天）”

### 后端 DB（外部）

- 默认必须分页（`limit/offset` 或等价游标）；任何可能返回大量数据的接口必须暴露“最大节点/最大行数”预算。
- 默认应启用时间窗口（例如 `30d`），除非用户/Agent 明确选择 `all/*`。
- 对“高风险全表扫描”条件必须显式标注风险，并优先建议改用：
  - 先插件候选集 → 再 DB 精筛；
  - 或增加更强的约束（ancestorId/pageId/tagId/timeRange）。
- 目标超时：单次查询 `≤30s`；但在 `better-sqlite3` 同步模型下无法可靠中断，**需要 worker/子进程隔离**才能实现硬超时（实现落地前仅能靠预算与分页降低风险）。

## 推荐的协作流程（给 Agent 的默认套路）

1. 先把用户意图“结构化成可检索线索”（关键词/范围/上下文 Rem）。
2. 若属于探索期定位：调用插件候选集搜索（`limit=20, timeout=3s`）拿到 Top‑K。
3. Agent 基于 `title + snippet` 选择少量候选（例如 3–5 个）。
4. 对候选 RemId 走后端 DB 工具做精筛/展开/分页，并在预算触顶时给出“下一步建议”。
