# 特性规格：批量拉取 + 冲突感知调度（消费链路调度与冲突提示）

**特性分支**: `[010-batch-pull-conflict-scheduler]`  
**创建日期**: 2026-01-25  
**状态**: Accepted  
**Accepted**: 2026-01-26  
**输入**: 用户描述：“在操作链路消费侧：保证稳定性与并发处理；当客户端长时间未打开导致队列堆积，系统需要能校验并提示同页/同 Rem 的写入/修改/删除冲突面；后台服务承载调度，在安全前提下尽可能快地消费。”

全局概念与术语裁决见：`specs/CONCEPTS.md`（WS Protocol v2、WriteFootprint/ConflictKey、in_flight usedKeys 等）。

## 依赖（必须先对齐）

- **003-ws-identity / 004-sync-reliability**：active worker + kick/quarantine 是消费链路的前提。  
- **013-multi-client-execution-safety（Accepted）**：attempt_id + CAS ack + ack 重试，用于消除“lease 回收 + 多客户端切换”导致的回执覆盖/状态回滚风险；本 spec 的 batch pull / scheduler 基于 013 的不变量实现。  

## 用户场景与测试（必填）

### 用户故事 1 - 客户端在线时：更少 WS 往返、更高吞吐（优先级：P1）

作为用户/智能代理，我希望在 RemNote 插件在线时，队列消费的吞吐更高、延迟更低，避免“每个 op 都要一次 `RequestOp`/`OpDispatch` 往返”的额外开销。

**独立可测**：在队列中放入 N 条互不冲突的 ops，插件以固定 `syncConcurrency` 运行，确认：

- WS 消息次数显著减少（批量拉取生效）。
- 消费完成时间明显下降或至少不劣于现状。

**验收场景**：

1. **给定** 队列中存在 100 条互不冲突的 ops，**当** 插件启动一次同步消费循环，**那么** WS 的“拉取消息”次数应从 ~100 次下降到 ~`ceil(100 / batchSize)` 次（不要求精确，但必须可解释）。

---

### 用户故事 2 - 客户端离线后回归：尽可能并发，但不扩大冲突面（优先级：P1）

作为用户/智能代理，当客户端长时间离线、队列积压几十个甚至更多操作时，我希望系统恢复消费后：

- 能尽可能并发地执行“互不冲突”的操作；
- 对于冲突操作，不要盲目并发扩大风险（例如同 Rem 的 update/delete；同 parent 的 children 结构变更），而应尽量被调度为安全顺序，或在同一批次中避免同时派发。

**独立可测**：构造 50 条 ops，包含多个“冲突簇”（同 `remId`、同 `parentId`），确认后台服务的调度会优先派发一批互斥键不重叠的 ops，冲突簇被拆散到后续批次。

**验收场景**：

1. **给定** 队列中有 10 条操作都写同一个页面（同 `children:<pageId>`），且另有 40 条操作分散在其他页面，**当** 启动一次消费，**那么** 系统应优先并发消耗那 40 条（不冲突），并把 10 条冲突簇按顺序消耗或拆分到后续批次。

---

### 用户故事 3 - 冲突校验与安全提示：让“冲突面”可见且可操作（优先级：P1）

作为用户/智能代理，当队列积压且存在潜在冲突（同页/同 Rem 的写入/修改/删除）时，我希望系统能在消费前/消费中产出可读且可机读的“冲突面报告”，用于：

- 判断风险（是否会覆盖/打架/造成结构混乱）
- 指导行动（例如：先停同步、拆分队列、取消某些 txn、或接受串行执行）

**独立可测**：给定一个包含明显冲突的 pending 集合，`agent-remnote queue conflicts`（或等价入口）能稳定输出：

- 冲突簇（按冲突键分组）的摘要
- 每簇影响的 op 数/txn 数
- 代表性 op 的类型与目标 Rem/页面线索

**验收场景**：

1. **给定** 队列中存在对同一个 `remId` 的 `update_text` 与 `delete_rem` 混在一起，**当** 查询冲突报告，**那么** 报告必须把该 `remId` 列为高风险冲突，并提示可能后果（例如“先删后改会失败 / 可能需要人工复核”）。

---

### 用户故事 4 - 协议升级（forward-only）：不做向后兼容，版本不匹配直接失败（优先级：P2）

作为维护者，我希望该优化以“向前兼容（forward-only evolution）”方式升级 WS 协议：

- 不保留旧的 `RequestOp` 单条拉取路径；升级后只支持 `RequestOps` / `OpDispatchBatch`
- 插件与后台服务的协议版本必须显式对齐（例如在 `Register` 中携带 `protocolVersion`）；不匹配时必须快速失败并给出可诊断信息（含 `nextActions`）
- 协议必须包含“派发尝试标识”（attempt token），以确保回执只确认当前派发（依赖 `specs/013-multi-client-execution-safety`）

**独立可测**：用旧插件（或模拟旧客户端）发送 `RequestOp`，后台服务应返回明确错误并拒绝继续处理；用新插件发送 `RequestOps`，能稳定拉取并消费。

---

### 用户故事 5 - 链路可诊断：队列库不可写/不可用时可行动（优先级：P2）

作为用户/智能代理，当写入链路在“入队阶段”失败（例如队列库不可写、路径错误、权限问题）时，我希望命令行工具返回足够可行动的信息，让我能快速修复并重试，而不是停留在模糊的“不可用”。

**独立可测**：人为把 `queue.sqlite` 指向一个只读目录或设置为只读文件，确认写入命令会：

- 给出稳定错误码（例如 `QUEUE_UNAVAILABLE`）
- 返回 `db_path` 与底层 sqlite 错误信息（例如 `attempt to write a readonly database`，表示“尝试写入只读数据库”）
- 给出可执行的 `nextActions`（例如检查权限/覆盖 `--queue-db`/设置 `REMNOTE_QUEUE_DB`/运行 `doctor`）

**验收场景**：

1. **给定** `queue.sqlite` 不可写，**当** 执行任意写入命令（`write/apply/replace`），**那么** 命令行工具必须输出可行动的错误信息（包含 `db_path` 与原始 sqlite 错误），并提供至少一条可复制执行的 `nextActions` 命令建议。

---

### 边界情况

- 队列超大（>10k ops）：冲突分析与调度不能扫描全表导致卡顿，应有上限与降级策略。
- `replace_selection_with_markdown` 这类依赖 selection/portal 的操作：冲突键推导要保守（宁可串行）。
- 目标 Rem 已不存在（先删后改）：冲突报告应能提示 “可能失败/需要人工确认”。
- 插件断线/崩溃：租约回收与重试不应造成无限重复执行或大规模重复创建。
- 队列 DB 不可写/只读：写入链路应在入队阶段快速失败，并给出可行动的诊断与修复建议。

## 需求（必填）

### 功能性需求

- **FR-001**: 系统必须在 WS 协议中新增“批量拉取”能力（例如 `RequestOps` / `OpDispatchBatch`），用于一次拉取多条可执行 ops。
- **FR-002**: 系统必须以 forward-only 方式升级协议：升级后仅支持 `RequestOps` / `OpDispatchBatch`；必须引入显式协议版本校验（例如 `Register.protocolVersion`），版本不匹配时必须快速失败并输出可诊断错误与 `nextActions`。
- **FR-003**: 后台服务必须引入“调度器”来挑选一批 ops 派发给插件执行端，并在派发时写入队列锁信息（claim/in_flight/lease）。
- **FR-004**: 调度器应该尽可能最大化并发：在同一个批次中尽量选择“冲突键集合不相交”的 ops（见 `data-model.md` 的 `ConflictKey`）。
- **FR-005**: 调度器必须遵守现有队列顺序语义：同一 txn 内的 ops 仍必须顺序执行，且同一 txn 在同一时刻最多一个 op 为 in_flight。
- **FR-006**: 系统必须提供冲突面报告的命令行入口（建议：`agent-remnote queue conflicts`），输出包含：
  - 按冲突键分组的摘要（op 数、txn 数）
  - 代表性 op 信息（类型、目标 Rem/页面线索）
  - 可执行的 `nextActions`（例如 inspect 某些 txn / 继续串行 / 建议拆分策略）
- **FR-007**: 当冲突面达到阈值（例如 backlog>阈值且存在高风险簇），系统应该在 `daemon sync` 或等价的触发点输出/返回安全提示（默认不强制阻断，除非用户显式开启严格模式）。
- **FR-008**: 批量拉取必须具备背压：插件请求的 `maxOps` 应与其空闲并发槽位匹配；后台服务必须对单次响应的 ops 数与字节量做上限约束，避免大帧导致卡顿或断线。
- **FR-009**: 租约策略必须与批量/并发匹配：默认租约时间应足够覆盖最慢 op 的执行，并提供可配置项或按 op 类型动态策略（见 `plan.md`）。
- **FR-010**: 系统必须增强写入链路的错误可诊断性，并支持 write-first：写入命令不要求单独的“事前检查”，应直接尝试入队；当队列 DB 不可用/不可写时，必须返回稳定错误码、`db_path`、底层 sqlite 错误信息，并提供 `nextActions`（可复制执行的英文 CLI 命令）指导修复与重试。
- **FR-011**: `doctor` 命令应该覆盖“队列 DB 可写性检查”，并在失败时给出明确修复建议（覆盖路径/权限/同用户运行等）。
- **FR-012**: `OpDispatchBatch.ops[]` 中的每个 op 必须携带 `attempt_id`；`OpAck`/`AckOk` 必须携带并校验 `attempt_id`（依赖 013；用于避免 stale ack 覆盖新派发）。
- **FR-013**: 冲突感知调度器在选择批次时，必须把“当前全局所有 `in_flight` ops 的冲突键”加入 `usedKeys`（避免 active worker 迁移后跨实例并发冲突写入）。

### 非功能性需求（性能与可诊断性）

- **NFR-001**: 调度器在“队列积压 10k”场景下仍必须保持可用：单次批量挑选的 DB 扫描范围需有上限（例如只扫描最前 200/500 个可执行 ops），并在超时/超预算时降级到简单策略（循环 `claimNextOp`）。
- **NFR-002**: 所有新输出必须可机器解析（稳定的 `--json` 封装输出），并包含足够诊断信息（例如批次大小、跳过原因、冲突键摘要）。
- **NFR-003**: 系统应该提供“同版本的快速降级开关”（仅用于开发期排障）：允许一键关闭“冲突感知调度”并退化为简单批量拉取（仍使用新协议）；不提供旧协议的兼容路径。
- **NFR-004**: 写入/调度相关输出应该自带 `nextActions`（英文命令），例如 `agent-remnote queue inspect --txn ...` / `agent-remnote queue progress --txn ...`，以便智能代理以“write-first”方式直接执行写入并自动闭环验证与排障。

### 待决问题（已记录，未裁决）

- 冲突提示的默认策略：仅提示不阻断 vs 阈值阻断（需 `--force`）？
- 冲突键的粒度：只做到 rem/children 级别，还是进一步引入 `pageId` 维度并做更友好展示？
- 是否引入批量回执（`OpAckBatch`）作为第二阶段优化（减少回执消息数）？
- 是否为写入类命令增加 `--wait/--timeout-ms`（或单独的 `queue wait`）以减少“入队后再手动 inspect”的步骤？

## 成功标准（必填）

### 可量化指标

- **SC-001**: 在“100 条互不冲突 ops”场景下，WS 拉取往返次数降低至少 80%（批量拉取生效证据）。
- **SC-002**: 在“50 条 ops、包含 3 个冲突簇”场景下，调度器能显著提高并发：非冲突 ops 的完成时间不劣于现状，并在日志/统计中可解释地展示“冲突导致的跳过/串行”。
- **SC-003**: 对同 rem 的 delete+update 混排等高风险冲突，命令行工具能在一次查询中产出可读报告，并给出可执行的 `nextActions`。
- **SC-004**: 在队列 DB 不可写场景下，写入命令能在一次失败输出中给出可行动的修复建议，使用户可在 1-2 次尝试内恢复入队能力。
